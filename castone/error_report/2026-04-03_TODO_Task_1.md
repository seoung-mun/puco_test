# Mayor 배치 입력 계약 재설계 문서
## 목표
- 사람 플레이어의 Mayor(시장) 배치 UX는 **토글 방식**을 유지한다.
- 엔진(`PuCo_RL`)이 요구하는 **순차 액션 계약**은 백엔드가 흡수한다.
- 프론트와 백엔드의 책임을 명확히 분리한다.
- 봇은 기존의 **순차 액션 경로**를 그대로 유지한다.
- 최종적으로 Mayor 입력 계약의 단일 책임 위치를 백엔드로 고정한다.

---

## 1. 현재 문제 정의

현재 Mayor 배치 로직은 아래가 섞여 있다.

- 사람 UI는 토글/최종 계획 기반에 가깝다.
- 엔진은 현재 커서/슬롯 기준의 순차 액션 모델을 사용한다.
- 일부 프론트/백엔드/legacy 경로가 서로 다른 계약을 가정하고 있다.

그 결과 발생하는 문제:
1. 프론트가 엔진 세부 계약(현재 슬롯, amount 액션, skip 규칙)을 너무 많이 알아야 한다.
2. 사람 UX와 엔진 액션 모델이 직접 맞지 않는다.
3. 책임 위치가 애매해서 버그가 생기면 프론트/백엔드 중 어디서 고쳐야 하는지 불명확하다.
4. 사람과 봇 입력 경로가 섞여 설계가 복잡해진다.

핵심 결론:
- 문제의 본질은 "토글 UI냐 순차 UI냐"가 아니라
- "사람의 최종 배치 의도를 엔진 순차 액션으로 누가 정확히 변환하느냐"이다.

---

## 2. 최종 설계 원칙

### 2.1 프론트 책임
프론트는 **사용자 의도(plan)** 만 표현하고 전송한다.

프론트가 책임지는 것:
- 토글 UI 렌더
- 슬롯별 선택/해제
- 최종 배치 계획 생성
- 최종 제출 버튼
- 서버 오류 시 최신 상태 재동기화

프론트가 책임지지 않는 것:
- 엔진 현재 커서
- 순차 액션 인덱스(예: 69~72)
- skip 타이밍
- 슬롯 순회 규칙
- 내부 amount 액션 변환

즉 프론트는 엔진 액션을 몰라도 된다.

### 2.2 백엔드 책임
백엔드는 사람 플레이어의 최종 배치 계획을 받아
**엔진 순차 액션으로 변환 + 검증 + 실행** 한다.

백엔드가 책임지는 것:
- Mayor phase 검증
- actor 검증
- 최종 배치 계획 검증
- stable slotId -> 엔진 슬롯 매핑
- 엔진 순차 액션 목록 생성
- 여러 엔진 step 실행
- 실패 시 명확한 에러 반환
- 성공 시 최신 GameState 반환

### 2.3 엔진 책임
엔진은 기존대로 순차 액션만 처리한다.
엔진 계약 자체는 바꾸지 않는다.

### 2.4 봇 책임
봇은 기존대로 순차 액션/마스크 기반 경로를 유지한다.
봇은 사람용 토글 계획 변환기를 사용할 필요가 없다.

정리:
- 사람 입력 경로: 토글 UI -> plan -> backend orchestrator -> sequential actions
- 봇 입력 경로: bot policy -> sequential action directly -> process_action()

---

## 3. 핵심 아키텍처 결론

### 최종 방향
- 사람 Mayor 배치는 **토글 UI 유지**
- 프론트는 **최종 배치 결과만 한 번에 제출**
- 백엔드에 **별도 orchestration 서비스 파일** 신설
- 이 서비스가 엔진 계약으로 변환
- 봇은 기존 sequential 경로 유지

### 별도 서비스 파일 위치
추천 파일:
- `backend/app/services/mayor_orchestrator.py`

이유:
- `game_service.py` 비대화 방지
- 책임 분리 명확
- 테스트 단위 분리 가능
- Mayor 관련 변환 로직을 한 곳에 집중 가능

## 4. 슬롯 식별 방식

### 4.1 stable slotId를 사용한다
프론트는 배열 인덱스가 아니라 의미 기반의 stable slotId를 사용한다.

예시:
- `island:quarry:0`
- `island:corn:1`
- `city:small_market:0`
- `city:large_market:0`

### 4.2 배열 인덱스를 직접 쓰지 않는 이유
배열 인덱스 기반의 문제:
- 렌더 순서 변화에 취약
- 프론트와 백엔드가 동일 정렬을 강하게 가정해야 함
- 디버깅이 어렵고 의미가 약함

stable slotId의 장점:
- 의미가 명확
- 프론트/백엔드 책임 분리 쉬움
- 로그와 테스트가 쉬움
- serializer가 동일 slotId를 내려주면 프론트는 그대로 사용 가능

### 4.3 slotId 생성 책임
slotId는 프론트가 임의 조합하지 말고,
**백엔드 serializer가 state 안에 포함해서 내려주는 방식**을 권장한다.

예시 응답 일부:
```json
{
  "slotId": "city:small_market:0",
  "buildingType": "small_market",
  "occupied": 0,
  "capacity": 1
}
````

## 5. 프론트 입력 모델

### 5.1 프론트는 최종 결과만 제출

프론트는 변경 이벤트 스트림(+1, -1, toggle delta)이 아니라
**최종 배치 결과(plan)** 를 제출한다.

권장 형식:

```json
{
  "placements": [
    {"slotId": "island:quarry:0", "count": 1},
    {"slotId": "city:small_market:0", "count": 1}
  ]
}
```

### 5.2 제출 방식

* 사용자가 토글 UI에서 배치를 다 정함
* 마지막에 확인 버튼 클릭
* 그 시점에 `mayor-distribute` 호출

권장하지 않는 방식:

* 토글할 때마다 서버 반영
* 부분 변경(delta)만 서버 전송

이유:

* 엔진 순차 모델과 충돌하기 쉬움
* 중간 상태 desync 위험 증가
* UX와 구현이 동시에 복잡해짐

## 6. API 표준 계약

### 6.1 표준 공개 API

사람 Mayor 입력의 표준 외부 API는 다음으로 통일한다.

`POST /api/puco/game/{gameId}/mayor-distribute`

### 6.2 요청 예시

```json
{
  "placements": [
    {"slotId": "island:quarry:0", "count": 1},
    {"slotId": "city:small_market:0", "count": 1}
  ]
}
```

### 6.3 성공 응답

* 최신 `GameState`
* 최신 `action_mask`
* 필요 시 최신 revision/meta

### 6.4 실패 응답

예시:

```json
{
  "detail": "Invalid mayor distribution: assigned colonists exceed available count."
}
```

또는

```json
{
  "detail": "Mayor distribution is stale. Please refresh state."
}
```

### 6.5 legacy / 내부 API

사람용 외부 계약은 `mayor-distribute`로 고정한다.
기존 `mayor-place`, `mayor-finish-placement`, `mayor-place-colonist` 류는

* 내부 구현 전용
* 또는 점진적 폐기 대상
  으로 정리한다.

즉 프론트는 sequential API를 직접 알지 않도록 한다.

## 7. Mayor Orchestrator 설계

### 7.1 추천 인터페이스

파일:

* `backend/app/services/mayor_orchestrator.py`

예상 함수:

* `validate_distribution_plan(engine, actor_id, plan)`
* `translate_plan_to_engine_steps(engine, plan)`
* `apply_distribution_plan(game_id, actor_id, plan)`

### 7.2 처리 흐름

1. 현재 game이 Mayor phase인지 확인
2. 현재 actor가 인간 플레이어이며 현재 차례인지 확인
3. 제출된 placements의 slotId들이 현재 state 기준 유효한지 검증
4. count 합계가 현재 배치 가능한 colonist 수를 넘지 않는지 검증
5. 현재 엔진 커서/slot 순서 기준으로 순차 액션 생성
6. 내부적으로 여러 엔진 step 실행
7. 최종 상태 직렬화
8. Redis/WS sync
9. 최신 상태 반환

### 7.3 중요한 원칙

* 이 서비스는 **엔진 액션 인덱스의 유일한 번역기**여야 한다.
* 프론트가 같은 번역을 중복 구현하면 안 된다.

---

## 8. GameService와의 통합

### 8.1 권장 방식

`GameService`는 Mayor plan을 직접 해석하지 말고 오케스트레이터에 위임한다.

예시 개념:

```python
if decision_type == "mayor_distribution":
    return mayor_orchestrator.apply_distribution_plan(game_id, actor_id, plan)
```

### 8.2 이유

* `GameService`는 전체 흐름 조정
* `MayorOrchestrator`는 Mayor 전용 변환 책임
* 역할 분리가 선명해짐

---

## 9. 검증 정책

### 9.1 최종 검증 책임

최종 검증 책임은 백엔드에 둔다.

프론트는 UX용 사전 제한만 할 수 있다:

* 남은 colonist 수 초과 금지
* 불가능한 슬롯 토글 제한

하지만 최종 진실은 백엔드다.

### 9.2 백엔드가 반드시 검증할 것

* 현재 phase가 Mayor 관련인지
* actor가 현재 행동 가능한 human인지
* placements의 slotId가 현재 상태에 존재하는지
* count 총합이 허용 범위 안인지
* 이미 가득 찬 슬롯에 과배치하지 않는지
* 최신 상태와 plan이 충돌하지 않는지

---

## 10. stale state 처리

### 10.1 정책

제출 시 최신 상태와 충돌하면
**백엔드 reject + 프론트 재동기화** 로 처리한다.

### 10.2 이유

토글 UI는 사용자가 상태를 보고 계획을 세우는 구조라,
중간에 상태가 바뀌면 plan이 오래된 정보일 수 있다.

### 10.3 확장 포인트

추후 필요하면 요청에 `baseRevision` 또는 state hash를 추가할 수 있다.

예:

```json
{
  "baseRevision": 42,
  "placements": [...]
}
```

초기 구현에서는 생략 가능하지만,
멀티플레이 안정성을 높이려면 향후 도입 권장.

---

## 11. 봇 처리 정책

### 11.1 결론

봇은 사람용 번역기를 사용하지 않는다.

### 11.2 이유

봇은 이미 순차 action/mask 기반으로 동작한다.
사람용 번역기는 "토글 UI의 최종 결과를 엔진 계약으로 바꾸기 위한 계층"이지,
모든 actor가 공유해야 하는 계층이 아니다.

### 11.3 최종 구조

* human only: `mayor-distribute` + orchestrator
* bot: 기존 `BotService` sequential 경로 유지

이렇게 입력 경로가 달라도 문제 없다.
최종적으로 엔진에 들어가는 action contract만 일치하면 된다.

## 12. 프론트/백엔드 책임 분리 최종표

### 프론트

* 토글 UI 제공
* slotId 기반 plan 생성
* 최종 제출
* 서버 에러 수신 후 UI 갱신
* 엔진 액션 인덱스는 몰라도 됨

### 백엔드

* plan 검증
* slotId -> 엔진 슬롯 매핑
* sequential action 변환
* 여러 step 실행
* 최종 상태 sync 및 반환

### 엔진

* 기존 sequential action 실행
* action mask/phase/current slot 유지

### 봇

* 기존 sequential path 유지

## 13. 구현 우선순위

### 1단계

* serializer에 stable slotId 추가
* 프론트 토글 상태를 slotId 기반으로 전환

### 2단계

* `mayor_orchestrator.py` 신설
* validate / translate / apply 구현

### 3단계

* `POST /mayor-distribute` API 추가
* `GameService`에 연결

### 4단계

* 프론트 `confirmMayorDistribution()`를 새 API에 연결
* 기존 사람용 sequential mayor 경로 제거 또는 우회

### 5단계

* 테스트 추가

---

## 14. 테스트 전략

### 14.1 백엔드 단위 테스트

* valid placement plan -> expected sequential action sequence
* colonist 초과 배치 -> reject
* invalid slotId -> reject
* stale phase -> reject
* wrong actor -> reject

### 14.2 백엔드 통합 테스트

* 사람이 Mayor 배치 plan 제출
* 여러 internal engine step 실행
* 최종 GameState와 action_mask가 정상 반영되는지 확인

### 14.3 프론트 테스트

* 토글 UI가 placements를 올바르게 생성하는지
* confirm 시 `mayor-distribute` 호출되는지
* 에러 응답 시 최신 상태로 재동기화되는지

### 14.4 회귀 테스트

* 봇 Mayor 경로가 기존처럼 동작하는지
* 사람 외 phase/action 경로가 영향 없는지

---

## 15. 최종 결론

이 설계의 핵심은 다음 한 문장이다.

**사람 Mayor 배치는 토글 UI를 유지하되, 프론트는 stable slotId 기반 최종 배치 계획만 보내고, 백엔드의 별도 Mayor Orchestrator가 이를 엔진 순차 액션으로 변환하고 적용한다. 봇은 기존 sequential 경로를 그대로 유지한다.**

즉:

* 사람 UX는 유지
* 엔진 계약은 유지
* 책임 분리는 강화
* 변환 책임은 백엔드 한 곳으로 집중

이 방향으로 구현해줘.

# Puerto Rico Multiplayer & MLOps Architecture Design

## 1. 아키텍처 개요 (Overview)
사용자 환경 구성을 바탕으로 설계된 전체 아키텍처입니다:
- **AWS 게임 서버 (단일 서버)**: 웹소켓 매니저를 통해 여러 유저의 세션(Room)을 관리하고, 코어 게임 엔진(`PuertoRicoGame`) 상태를 유지합니다.
- **학습용 GPU 서버 (분리된 서버)**: 게임 서버에서 생성된 플레이 로그 데이터를 수집하여 RL(강화학습) 파이프라인을 돌립니다.

---

## 2. 봇 추론(Inference) 시스템 구축 옵션 및 추천

게임 룸에는 반드시 1개 이상의 AI 봇이 참여합니다. 이 봇들의 '다음 행동(Action)'을 어디서 계산할지에 대한 2가지 접근 방식입니다.

### 옵션 A: 게임 서버 내장 추론 (Local Inference) - **[권장]**
게임 서버(AWS)의 백엔드 프로세스 메모리 내에서 ONNX Runtime 파이썬 패키지를 이용해 직접 봇 모델을 실행합니다.
- **장점**: 
  - 네트워크 지연(Latency) 통신 비용 제로. 가장 빠른 반응 속도.
  - 시스템 단순화 (API 통신 실패 등 예외 처리 불필요).
- **단점**: 
  - AWS 게임 서버의 CPU/Memory 자원을 일부 소모 (다만 턴제 보드게임 추론은 매우 가벼움).
- **사용자 환경 매칭도**: **최상.** 실시간성을 요구하지 않는 턴제 보드게임의 소규모 행렬 연산은 CPU만으로도 10ms 이내에 충분히 처리됩니다. 

### 옵션 B: 별도 추론 서버 연동 (Remote API Inference)
학습용 GPU 서버 혹은 추론용 추가 서버에 FastAPI/BentoML 기반의 API 서버를 띄워, 게임 서버가 API를 호출해 봇 턴을 받아옵니다.
- **장점**: 
  - 게임 서버와 AI 연산의 완벽한 리소스 분리.
  - 대규모의 무거운 딥러닝 모델 적용 시 유리.
- **단점**: 
  - HTTP 통신 오버헤드로 인한 Latency 증가, 에러/타임아웃 핸들링 로직 추가 구조 복잡성.
- **사용자 환경 매칭도**: 낮음. 훈련(Training)에는 GPU가 필요하지만, 턴제 게임 추론은 가볍기 때문에 학습용 서버 자원을 상시 추론 API용으로 켜두는 것은 비용 낭비일 수 있습니다.

> **건축가 추천 (Recommendation)**: 
> 훈련(Training)은 GPU 서버에서 비동기 배치(Batch)로 진행하되, 완료된 가벼운 모델 가중치 파일(`.onnx` 또는 `.pt`)을 AWS 게임 서버로 폴링/동기화 시켜 **옵션 A(내장 추론)**로 구동하는 것을 강력히 추천합니다.

## 3. 웹소켓 및 룸(Room) 관리 설계
`castone/backend/app/services/ws_manager.py` 구조를 응용합니다.
- **ConnectionManager**: `<game_id>` 단위로 Room을 생성 및 관리합니다.
- **State Registry**: 현재 싱글톤인 `manager` 객체 대신 `dict[game_id, GameManager]` 구조의 메모리 풀을 사용합니다.
- **Event Flow**:
  1. 클라이언트(유저) 접속 및 `action` 전송.
  2. `GameManager(game_id)`가 로직 처리 및 상태 업데이트.
  3. 상태 변화 발생 시 `ConnectionManager`를 통해 해당 Room 유저 전원에게 새로운 JSON 브로드캐스트.
  4. 다음 턴이 봇인 경우: `Local Inference` 모듈을 비동기로 호출 (2초 딜레이 UX 유지) -> 다시 결과 브로드캐스트.

---

## 4. 데이터 파이프라인 (학습 연결 고리)
강화학습 재학습을 위한 데이터 수집 로직입니다.
1. 게임 종료(`check_game_end_game`) 시점에 `GameManager`는 최종 승자 정보, 플레이 내역, 턴별 상태-행동(S-A) 로그를 **JSON 라인 포맷**으로 직렬화합니다.
2. AWS 게임 서버의 특정 디렉토리(`데이터 레이크`)에 적재합니다.
3. 주기적으로 학습 GPU 서버가 해당 디렉토리의 파일을 동기화(rsync 또는 Amazon S3 활용)하여 PPO/DQN 등 RL 파이프라인 학습 배치를 돌립니다.
4. GPU 서버가 새 모델(weights)을 뱉어내면 S3 등에 업로드하고, 게임 서버는 이를 다운로드 받아 Reload 합니다.

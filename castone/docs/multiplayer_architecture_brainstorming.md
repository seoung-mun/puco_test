# Puerto Rico Multiplayer Architecture: Understanding & Brainstorming

## 1. Understanding Summary (현재 이해도)

현재 백엔드는 싱글 턴 기반 또는 로컬 봇 매치에 맞춰진 **싱글톤(Singleton) 구조**로 구현되어 있습니다.

새로운 목표는 이를 **다중 접속 실시간 멀티플레이어 환경**으로 확장하는 것입니다:

게임의 데이터를 기반으로 봇을 강화학습으로 재학습 시킬예정이야

- **What**: 여러 명의 플레이어가 동시에 각기 다른 게임 룸(Room) 세션을 생성하고 참여할 수 있는 다중 게임 백엔드 구현.

- **Why**: 개별 유저의 로컬 플레이를 넘어, 실제 웹 서비스 형태의 대규모 동시 접속 환경(MMO/Room-based)을 지원하기 위함.

- **Who**: 사람 대 사람 로비 매칭 및 사람 대 봇(AI) 혼합 매칭을 즐기려는 일반 웹 유저들.
  -> 최소 봇이 1개 이상은 있을거야, 최대 3인플(사람1,봇2/사람2,봇1)

- **Key Constraints**:

- 현재 단일 JSON 파일(`game_state.json`, `lobby.json`)에 의존하는 상태 저장 방식을 세션별로 분리해야 함.

- 현재 폴링(Polling) 방식으로 동작하는 프론트엔드 통신을 WebSocket(또는 Redis Pub/Sub) 기반의 실시간 이벤트 동기화로 전환하거나 병행해야 함.

- **Explicit Non-Goals**:

- 클라이언트-사이드(프론트엔드) 코어 비즈니스 로직 작성 (모든 규칙과 상태는 백엔드가 통제).

## 2. Assumptions (사전 가정 사항)

향후 설계를 진행하기 위해 다음과 같이 가정합니다. 이 중 사실과 다른 부분은 수정이 필요합니다.

1. **상태 관리 (State Management)**: 현재 단일 `manager` 인스턴스 대신, `GameManager` 인스턴스를 메모리(또는 Redis/DB)에 `game_id` 별로 여러 개 유지관리하는 Registry 구조가 필요하다.

2. **저장소 (Storage)**: 최소한 초기에는 로컬 파일(디렉토리 분리)이나 Redis를 활용하며 추후 PostgreSQL 같은 RDBMS로 전환 가능하다.

3. **통신 방식 (Communication)**: `castone/backend/app/services/ws_manager.py`에 구현되어 있는 WebSocket + Redis Pub/Sub 패턴을 이 프로젝트에 이식하여 실시간 상태 동기화를 구현한다.

4. **ML/봇 통합 (AI Integration)**: MLOps 워크플로우에 따라 여러 방에서 동시에 AI 봇 턴 처리가 발생하더라도 병목이 없도록 (가능하다면 Celery나 분산 큐를 활용하여) 설계해야 한다.

## 3. Open Questions (해결해야 할 질문)

1. **서버 스케일링 필요성**: 단일 백엔드 서버에서만 모든 룸을 처리할 예정입니까, 아니면 미래에 여러 대의 백엔드 컨테이너(서버)가 로드밸런싱 하에서 동작하는 분산 환경을 염두에 두어야 합니까? (성능 및 DB 선택에 큰 영향을 미칩니다.)
   -> 게임 환경은 단일 백엔드 환경에서 처리할 예정이야, 모델 재 학습은 따로 gpu 서버를 빌릴 예정이라 그건 로직만 만들어줘

2. **봇 추론 시스템 분리**: 다수의 게임 룸에서 봇이 동작할 텐데, 봇의 행동 결정(추론)을 백엔드 메모리에서 직접 계산할지, 아니면 MLOps 파이프라인(별도 AI 추론 서버/API)을 통해 분리할 계획인지 궁금합니다.
   -> 각각의 장단점을 정리해줘 그리고 내 환경에서 가장 추천하는 방식을 추천해줘

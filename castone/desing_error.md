# Castone 봇 무반응 및 주지사 랜덤 배정 오류 심층 분석 보고서

**날짜:** 2026-04-02
**분석 대상:** 
1. 주지사 랜덤 배정 시 방장 소외 버그 (Inclusive Randomness)
2. 봇(Bot) 스케줄링 진입 실패 및 로그 부재 (Silence Failure)

---

## 1. 결함 A: 주지사 랜덤 배정의 정합성

### 🔴 현상
- (과거) 방장(`player_0`)이 첫 주지사 배정에서 계속 제외됨.
- (현재) 방장만 주지사가 되는 역버그 발생 중.

### 🔴 원인 추정
- `PuCo_RL` 엔진의 `reset()` 과정 중, 주지사가 정해진 후 특정 초기화 로직이 `governor_idx`를 건드리거나, `player_0`을 유효한 주지사 후보에서 누락시키는 논리적 결함 존재 가능성.

### ✅ 해결 전략 (허가 후 실행)
- `reset()` 반복 호출 대신, 엔진 초기화 직후 `game.governor_idx`를 `random.randint(0, num_players-1)`로 **강제 재설정**하고, 이에 맞춰 `current_player_idx`와 `agent_selection`을 수동 동기화하여 방장을 포함한 완전한 랜덤성 확보.

---

## 2. 결함 B: 봇의 침묵 및 로그 부재

### 🔴 현상
- 프론트엔드: "Bot 차례 대기 중" 고착.
- 백엔드: 어떠한 봇 관련 로그도 출력되지 않음.

### 🔴 원인 추정 (Type Mismatch)
- **문제 지점:** `game_service.py`의 `if str(next_actor).startswith("BOT_"):`
- **상세:** `room.players`에 저장된 값이 `UUID` 객체인 경우, `str(uuid)`는 `BOT_` 접두사가 없는 순수 ID 값만 반환함. 
- **결과:** 백엔드는 봇을 사람으로 오판하여 스케줄링을 생략하고, 프론트엔드는 본인 차례가 아니므로 무한 대기.

### ✅ 해결 전략 (허가 후 실행)
- **가시성:** `logger` 대신 `print(..., flush=True)`를 사용하여 도커/로컬 로그 출력 보장.
- **판별 로직 강화:** 플레이어 명단뿐만 아니라, 현재 `EngineWrapper`가 알고 있는 봇 인덱스 정보를 교차 검증하여 봇 여부를 판별.
- **환경 분리:** 로컬 터미널 분리 실행을 통해 실시간 `print` 디버깅 수행.

---

## 3. 디버깅 가이드 (로컬 터미널용)

1. **백엔드 환경 변수 설정:** 
   - `export PYTHONPATH=$PYTHONPATH:$(pwd)/PuCo_RL`
   - `.env` 파일 로드 확인
2. **실행:** `uvicorn app.main:app --reload`
3. **관찰:** 게임 시작 시 `print`로 출력되는 `next_actor`의 값과 타입을 대조.

---

## 4. 결론
현재 봇이 멈추는 가장 큰 이유는 **"봇을 봇이라고 알아보지 못하는 판별 로직의 한계"**로 보입니다. 주지사 랜덤화는 엔진의 초기 설정을 수동으로 보정하는 방식으로 해결해야 합니다.

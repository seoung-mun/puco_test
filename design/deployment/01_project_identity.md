# 01. Project Identity & Goals

**이 문서가 존재하는 이유**: Castone 프로젝트는 겉보기에 "웹 게임 서비스"처럼 보이지만, 실제 정체성은 **RL 실험 플랫폼**이다. 이 차이가 전체 배포 트랙의 의사결정을 좌우하기 때문에, 정체성을 명확히 박아둔다.

---

## 프로젝트 한 줄 정의

> **"Puerto Rico 보드게임의 강화학습 에이전트를 개선하고, 그 에이전트로 게임 밸런스를 데이터 기반으로 검증하는 실험 플랫폼"**

---

## 1차 목표 (Primary Goals)

이것이 성공의 진짜 기준이다.

### 1.1 RL 에이전트 성능 개선
- 새 PPO / Hierarchical PPO 체크포인트를 학습하고, 이전 버전 대비 승률 향상을 정량적으로 측정한다
- 측정 기준은 **"이전 프로덕션 모델 대비 eval tournament 승률 > 임계값(예: 55%)"**
- 이 측정을 자동화하는 것이 Phase 8 (D 단계)의 목적이다

### 1.2 게임 밸런스의 데이터 기반 검증
- 역할(주지사/선장/시장/상인/건설업자/생산자/금광업자) 선택 빈도가 균형적인가?
- 승리한 게임에서 공통적으로 구매된 건물이 있는가? ("OP 건물" 탐지)
- 게임 길이, 점수 분포, 결정적 턴의 특성은 무엇인가?
- 봇끼리 대전(self-play)과 사람 포함 대전 간 전략 차이는?

이 질문들에 대답하기 위한 **대시보드와 쿼리 가능한 데이터** 가 이 설계의 핵심 산출물이다.

---

## 2차 목표 (Secondary Goals)

1차 목표를 달성하는 과정에서 **수단이자 부수효과**로 발생한다.

### 2.1 과동아리 홍보 채널
- 동아리원들에게 "이거 내가 만든 RL 프로젝트야" 라고 보여줄 수 있는 공개 URL
- 접속 규모는 5~30명 예상, 홍보 피크에 일시적으로 더 많을 수 있음

### 2.2 인간 대전 데이터 수집
- 사람이 봇과 대전한 로그는 **RL 재학습의 귀중한 데이터**
- 사람의 수 패턴은 자기 대결(self-play)로는 얻을 수 없는 분포를 제공
- 이 데이터가 누적되어 다음 훈련 라운드의 입력이 됨 → **선순환 구조**

### 2.3 포트폴리오 어필
- 취업·이직 시 "이런 걸 만들었다"고 보여줄 수 있는 실물
- 단순 CRUD가 아니라 **MLOps·IaC·관측성까지 포함한 풀사이클** 이라는 점이 차별화
- 상세는 [15_portfolio_readme_checklist.md](./15_portfolio_readme_checklist.md)

## Non-Goals (의도적으로 하지 않는 것)

배포 초심자가 가장 많이 빠지는 함정은 **"이것도 필요하지 않을까?"** 인데, 아래 항목들은 **명시적으로 안 한다**.

| Non-Goal | 왜 안 하는가 |
|---|---|
| **고가용성 (HA), 다중 리전** | 동아리 규모에서 과투자. 99% 가용성이면 충분 |
| **관리형 DB (Cloud SQL / RDS)** | 월 비용 2~3배, 이 규모에서 가치 없음 |
| **Kubernetes / ECS / Cloud Run** | Docker Compose 한 방이 모든 면에서 더 적절함 |
| **상시 스테이징 환경** | VM 두 배 비용. PR CI가 스테이징 역할 |
| **Windows 네이티브 로컬 개발** | 팀원은 Mac/WSL2 사용. 배포 성공 시 자동 해소 |
| **상용 SLA (99.9%+)** | 포트폴리오엔 99%로 충분, SLA 수립 자체가 과투자 |
| **24x7 온콜** | 혼자 운영함. 다운 시 이메일/Telegram 알림 받고 아침에 대응 |
| **훈련 측 MLOps (W&B / MLflow 통합)** | PuCo_RL 폴더 수정 금지 제약. 훈련 코드에 instrumentation 못 넣음 |
| **봇 간 self-play 실시간 시청 기능** | 기능 과잉, 현재 범위 밖 |
| **유저 프로필/친구/채팅** | Puerto Rico 플레이에 불필요 |
| **결제/유료화** | 프로젝트 목적과 무관 |
| **다국어 UI** | 한국어만 지원. 동아리 대상 |
| **모바일 네이티브 앱** | 반응형 웹으로 충분 |

## 프로젝트 이해관계자

### 본인 (Seoung-mun)
- **역할**: 풀스택 개발, 배포, 운영, 데이터 분석 전담
- **환경**: Mac 로컬 개발
- **가용성**: 주당 ~10시간 프로젝트 할당 가능 (추정)
- **배포 경험**: 처음

### PuCo_RL 팀
- **역할**: 강화학습 훈련 코드 유지·보수, 새 체크포인트 생성
- **경계**: `PuCo_RL/` 폴더는 **다른 팀원 작업 영역**. 웹 측에서 **절대 수정 금지**
- **상호작용 지점**: GCS 모델 버킷에 새 `.pth` 업로드 + `metadata.json` 작성
- **기대**: 새 모델이 생기면 GCS에 올리기만 하면 웹 측이 자동 처리

### 동아리원
- **역할**: 유저, 데이터 제공자
- **기대**: 구글 로그인 한 번으로 바로 게임 플레이, 무료
- **사용 패턴**: 저녁/주말 비정기 접속

### 면접관·채용 리뷰어
- **역할**: 포트폴리오 평가자
- **기대**: README에서 빠르게 "이 사람이 뭘 할 줄 아는지" 파악
- **열람 시간**: README 2~3분 + 라이브 데모 5분 미만

## 이 설계가 다루지 않는 주변 트랙

Castone에는 여러 트랙이 있고, 이 설계는 **배포/MLOps 트랙만** 다룬다. 다른 트랙은 별도 문서가 있거나 병렬로 진행 중이다.

| 트랙 | 주체 | 상태 | 이 설계와의 관계 |
|---|---|---|---|
| 엔진 컷오버 (dual_mayor) | 본인 | 진행 중 | 독립적. 배포 전 엔진이 안정화되는 게 이상적 |
| `mayor` 페이즈 전략 전환 | 본인 | 보류 (engine 방향 확정 대기) | 배포 후로 미룸 |
| RL 훈련 코드 개선 | PuCo_RL 팀 | 외부 | 이 설계는 훈련 산출물(.pth 파일)만 소비 |
| 리플레이 UI 기능 | 본인 | **이 설계에 병합** (Phase 7) | TODO 3번 → 여기로 흡수 |
| 로그 구조 재설계 | 본인 | **이 설계에 병합** (Phase 6) | TODO 3번 → MLOps 데이터 스키마와 통합 |
| 배속/일시정지 등 UI | 본인 | 별도 | 작은 기능, 배포와 무관 |
